# Fable枠の使い方 — 「Opusに戻った日にFable級を引き出す」ための言語化リスト

作成: 2026-07-02。ユーザー方針の言語化:
**Fableの本命は、Fableが使える間に"賢さを仕組みに焼き付けて"、提供終了後のOpusでも同水準を再現できるようにすること。**
「賢い実行」に使うだけでは提供終了と同時に価値が消える。残るのは仕組み・文書・検証系だけ。

## 判断基準（毎作業で自問）

> この作業の成果は、**Fableがいなくなっても機能するか？**

- 機能する（設計書・検証スクリプト・RUNBOOK・テンプレ・フック）→ Fable期に優先してやる ✅
- 機能しない（その場の賢い判断・単発の実行）→ 仕組み化できないか先に考える ⚠️

## 公式「Prompting Claude Fable 5」の取込（2026-07-05・ユーザー提供の公式ドキュメントより）

出典: https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/prompting-claude-fable-5

**Fableが得意なこと（公式・Opus 4.8比）**: ①長期自律（数日規模のゴール駆動ラン・指示保持）②よく仕様化された複雑問題の一発正解 ③vision（技術画像・劣化画像に強い）④企業文書ワーク（分析・スプレッドシート・スライド）⑤コードレビュー/バグ発見の再現率 ⑥曖昧さのナビゲーション ⑦並列サブエージェントの信頼性。※攻撃的サイバー/バイオは分類器が拒否（→Opusフォールバック運用が正）。

**採用した原則（うちの実装先）**:
- 「行動できる情報が揃ったら行動（過剰計画禁止）」→ 夜間wrapperプロンプトに追加
- 「進捗報告は必ずツール結果と突合（証拠なき完了報告の禁止）」→ 同上（公式テストで捏造報告ほぼ根絶）
- 「依頼だけでなく理由を渡す」→ wrapperに「公開アカウント向け・正確性が信用を守る」の意図文を追加
- 「一番難しい問題から投げる」「メモリ表面を与える」「検証をサブエージェント/機械に分離」→ 既存運用と一致（verify系・memory・RUNBOOK）
- 「reasoning を応答に書き写させる指示は refusal（reasoning_extraction）を誘発」→ 全RUNBOOK/スキルを監査、該当なしを確認

**意図的に採用しないもの（判断記録）**:
- 「旧モデル向けの過剰に手順的なプロンプト/スキルはFableでは品質を下げるので削れ」→ **うちは削らない**。
  RUNBOOKの手順書型はOpusフォールバック互換のための設計（本プランの核心）。Fableには冗長でも実害が小さく、
  Opusには必須。Fable専用に最適化して提供終了後に壊れる方が損、という本プランの思想を優先する。
- send_to_userツール → ヘッドレス夜間はログ＋トースト＋スタジオが同役割を既に果たしている（不要）。
